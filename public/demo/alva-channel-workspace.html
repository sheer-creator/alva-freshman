定义

Alva Channel 是 Alva Agent 的长期上下文空间。它承载主会话、Task、Artifacts、Memory、Skill、Member、Connect IM 和 Settings。

每个用户初始拥有一个 Alva Agent Channel。之后可以从它平行分化出多个 Personal Topic Channel，也可以把 Telegram / Discord 群镜像成 Mirrored IM Group Channel。

Alva Agent Channel

Alva Agent Channel 是每个用户初始存在的默认 Channel。它承接 Web 全局 Ask、默认 Alva Agent 对话、未归属通知、未归属产物和 IM DM context。

GUI 示意 · 默认 Channel 主会话

基础规则：

  • 参与者是用户和 Alva Agent。
  • 默认 private。
  • 在 Channels section 中默认置顶。
  • 任意 Web 页面打开全局 Ask，默认进入 Alva Agent Channel。
  • 固定存在，不参与普通删除。
  • 可以创建 Personal Topic Channel。
  • 可以把任务、产物、旧 Chat 或 task summary 移动到 Personal Topic Channel。

页面 tabs：

Tab优先级内容
ChatP0Channel main session，默认进入的主对话
ArtifactsP0Channel 产物容器，分为 Playbooks、Automations、Files
TaskP1Task 列表、手动创建 task、task status、task session 入口
MemoryP1长期记忆、用户偏好、关键背景、重要决策
SkillP2默认 skill、临时 skill 使用记录、skill 开关

Chat

Alva Agent Channel 拥有一个 main session。用户在这个 main session 中持续对话，默认行为是 continue。

Web 全局 Ask、Agent 入口和 Channels section 置顶项都会进入这个 main session。Start fresh 只清理短期上下文窗口，保留 Artifacts、Task、Memory、Skill 和 Settings。

过渡期保留 New Chat 和 Chat History 作为 legacy 入口：

  • New Chat 不作为默认起手入口。
  • Chat History 不作为 Channel IA 核心入口。
  • 旧 Chat 可以保存为新 Channel。
  • 旧 Chat 可以移动到已有 Channel。
  • Task 未上线前，复杂指令先在 Channel main session 中执行，输出写入 Artifacts。

Task

Task 是挂载在 Channel 下的独立工作单元，本质是一个独立 session。

GUI 示意 · Task 是 Channel 下的子 session

创建方式：

  • 自动创建：用户在 main session 中提出复杂指令，Alva 判断需要 sub agent、长耗时工具调用或后台任务。
  • 手动创建：用户在 Task tab 点击创建 task。
  • IM 触发：Mirrored IM Group Channel 中的 @Alva、命令、reply、Group Admin 规则触发。

运行规则：

  • Task 属于一个 parent Channel。
  • Task 创建时读取 parent Channel 允许访问的 context、Artifacts、Memory、Skill。
  • Task 拥有独立 task session。
  • 用户可在 task session 内继续对话、补充信息、调整任务。
  • Task session 不污染 Channel main session 的短期上下文。
  • Task 的状态、进度、输出展示在 Task tab。
  • Task 产出的 Playbook 草稿、Automation、File 写入 Artifacts。
  • Task 完成后可把 summary、decision、memory candidate 回写到 parent Channel。
  • Task 未上线前，复杂指令在 Channel main session 中执行，输出写入 Artifacts。

Artifacts

Artifacts 是 Channel 的产物容器，包含三类：

GUI 示意 · Artifacts 产物容器

类型内容规则
PlaybooksChannel 关联、生成、使用过的 Playbook全局 Playbooks 保持独立对象库；Channel 内 Playbooks 归入 Artifacts
AutomationsChannel 触发、生成、订阅或运行的 AutomationAutomation 是 Feed 面向用户端的语言；展示状态、cadence、last run、pause / resume
Files文件、报告、source pack、图片、网页、策略草稿可回到 source task 或 source message

Automation 投递规则：

  • 创建 Automation 时可以选择消息投递的 Channel。
  • 默认投递到 Alva Agent Channel。
  • 可选投递目标包括 Alva Agent Channel、用户拥有的 Personal Topic Channel，以及用户有 Group Admin 权限的 Mirrored IM Group Channel。
  • Automation 的管理入口仍在 Artifacts / Automations 中；推送消息进入所选 Channel 的 Chat。
  • Phase 1 先支持单一投递 Channel；多 Channel 投递作为后续能力。

归属规则：

  • 主会话产生的产物写入当前 Channel Artifacts。
  • Task 产物写入 parent Channel Artifacts。
  • 用户可把 Alva Agent Channel 的 artifact 移动到 Personal Topic Channel。
  • Artifact access 控制具体文件、Playbook、报告、source pack 的读取和使用。
  • Automations 使用 Feed 产物作为数据来源；Files 需要补充对象归属、来源消息、权限和列表查询能力。

Memory

Memory 是 Alva 服务当前 Channel 的长期记忆。

GUI 示意 · Channel Memory 可追溯事实卡

当前现状：

  • Memory 是用户可见文件，不是隐藏模型状态；新用户会初始化 ~/memory/MEMORY.md 和 ~/memory/user.md。
  • user.md 承载用户身份、投资风格、知识水平和偏好模板；~/memory/ 下的主题文件承载长期背景，Agent 应先读 memory index，再按需读取相关文件。

默认处理：

  • main session 普通消息留在短期上下文。
  • 用户显式保存时写入 Memory。
  • Task 输出可以生成 summary、artifact 或 memory candidate。
  • Group Admin 规则命中时可以生成 task 或 memory candidate。
  • Alva 总结可以生成可追溯 memory candidate。

Channel 扩展：

  • Memory 需要从用户级扩展到 Channel 级，至少区分 user、personal_topic_channel、mirrored_im_group_channel 三种 scope。
  • 用户级 Memory 可作为基础画像被继承；Channel Memory 只保存当前主题或当前群组需要长期复用的信息。
  • Memory 必须支持查看、编辑、删除、source trace、scope 标记，以及 confidence 或 verification 状态。

Skill

Skill 是 Channel 级能力配置，Phase 3 落地。

规则：

  • 可按 Channel 设置默认 skill。
  • 单次对话或单次 task 可临时 override。
  • Skill tab 展示最近使用的 skill 和来源 task。

Member

Alva Agent Channel 是单人 Channel。

角色：

  • Owner: 管理 Channel 名称、visibility、memory、settings、归档状态。
  • Alva Agent: 组织、总结、检索、提醒、执行任务。

Connect IM

IM 连接规则：

  • Web 是主工作面。
  • Telegram / Discord DM 可作为 Alva Agent Channel 的外部入口。
  • IM DM context 默认归属 Alva Agent Channel。
  • Telegram / Discord group 进入 Mirrored IM Group Channel 模型。
  • 外部群不写入 Alva Agent Channel。
  • IM 连接不改变个人资产权限边界。

Settings

Settings 管理 Channel 级配置：

  • Profile: Channel 名称、description、icon。
  • Visibility: private / member-gated / public read。
  • Notifications: 频道通知、IM 推送、digest。
  • Memory policy: 自动保存策略、source trace、删除规则。
  • Default skill: Channel 默认 skill。
  • Connected IM: Telegram / Discord endpoint 绑定状态。
  • Archive / Delete: 归档、恢复、删除。

Personal Topic Channel

Personal Topic Channel 是从 Alva Agent Channel 平行分化出来的单人主题 Channel。

GUI 示意 · Personal Topic Channel 创建

基础规则：

  • 参与者是用户和 Alva Agent。
  • 默认 private。
  • 位于 Channels section，排在 Alva Agent Channel 下方。
  • 用户主动创建入口只有两个：侧边导航栏 Channels section 的 + New Channel，以及 Alva Agent Channel 中的自然语言创建请求。
  • 每个 Channel 创建完成后必须有 name 和 description。
  • 主体验是在同一主题内持续对话和推进任务。

创建、修改和删除：

  • 通过 + New Channel 创建时，用户可直接填写 name 和 description，也可只写主题，由 Alva 生成初始 name 和 description。
  • 通过 Alva Agent Channel 创建时，用户用自然语言说明要长期跟进的主题；Alva 生成 Channel name、description 并创建。
  • Playbook、Skill、Automation 不能直接创建 Channel；它们只能在 Channel 创建后被加入 Artifacts、Skills 或相关 context。
  • Owner 可修改 Channel 的 name 和 description。修改不改变主对话、Artifacts、Task、Memory 的归属。
  • Owner 可删除 Personal Topic Channel。删除后该 Channel 从 Channels section 移除，并停止作为默认路由目标。
  • 删除 Channel 不直接删除 Playbooks、Automations、Files 等独立对象；这些对象保留自己的生命周期和访问规则。
  • 如果 Channel 下有 running Task 或 active Automation，删除前需要确认处理方式：停止、移出或保留为独立对象。
  • Alva Agent Channel 是默认置顶 Channel，不支持删除。

Onboarding Preset Channel

平台可以在新用户 onboarding 时预设一个或多个 Personal Topic Channel，用于把用户直接带入一个已经运转的主题空间。

GUI 示意 · Onboarding preset channel

触发规则：

  • 预设 Channel 由平台按注册渠道创建，不占用用户主动创建入口。
  • 预设依据只使用注册渠道，例如 referral link、campaign link、KOL invite link 或 partner landing page。
  • 新用户首次进入 Alva 时，可以默认落到最相关的预设 Channel，而不是 Alva Agent Channel 的空白主对话。
  • 预设 Channel 仍然归用户所有，默认 private，用户可改名、改 description、暂停 Automation 或删除 Channel。

内容规则：

  • 预设 Channel 必须有明确 name 和 description，说明这个 Channel 追踪什么主题。
  • 平台可以为预设 Channel 添加若干 Automations 和 Playbooks。
  • Automations 可以先回填真实历史输出，让用户一进来就看到这个 Channel 已经推送过哪些市场消息、摘要、提醒和 source。
  • 预设 Automations 支持快速参数填写，用户补充 tickers、watchlist、KOL handles、推送频率、投递 Channel 等必要参数后，就能变成自己的 Automation。
  • 预设 Channel 可以展示特化的 setup Widget，用于填写 Automation 的必要个性化参数。
  • setup Widget 应直接服务当前预设 Automation，例如选择 tickers、上传 watchlist、填写 KOL handles、选择 cadence、选择投递 Channel、连接 Telegram。
  • 用户完成 setup Widget 后，系统把预设 Automation 复制或激活为用户自己的 Automation；历史示例消息继续作为 sample，不伪装成用户自己的历史。
  • 预设 Playbooks 用作示例和起点，用户可直接使用、remix，或把其中的 Automation 改成自己的版本。
  • 历史消息必须来自真实 Automation run、真实公开 source 或明确标记的 demo sample，不能伪装成用户自己的历史对话。
  • 预设 Channel 的目标是让用户理解“如何设置一个专属于自己的推送”，因此每条 Automation 消息都应能回到对应 Automation 的规则、cadence、source 和开关。

示例：

  • 从 AI infra / KOL campaign 注册的用户，进入 KOLs · AI Digest Channel。
  • Channel 内预置 KOL-AI-Digest、Serenity-New-Call-Alert 等 Automations，以及一个可 remix 的 AI infra Playbook。
  • Chat 里展示过去若干条真实 digest push；Artifacts 里展示 Automations、Playbooks、相关 files 和 source list。
  • 用户可以通过 setup Widget 填写参数，或继续对话：“只保留 semis 相关信号”、“改成每天盘前推送”、“把这个同步到 Telegram”。

继承与隔离：

  • Chat: 拥有独立 main session。
  • Artifacts: 拥有独立 Artifacts，分为 Playbooks、Automations、Files。
  • Task: 复用 Channel Task 模型；Task 属于当前 Topic Channel。
  • Memory: 只保存该主题的偏好、研究路线、历史判断、已验证结论、已推翻结论。
  • Skill: 可继承 Alva Agent Channel 默认 skill，也可设置 topic-specific skill。
  • Member: Phase 1 只支持 Owner + Alva Agent。
  • Connect IM: 默认 Web-first；Telegram / Discord 的 Channel 级连接按下方规则逐步开放。
  • Settings: 独立 name、description、visibility、notification、memory policy、default skill。

Connect IM

Personal Topic Channel 的 IM 连接不替代 Alva Agent Channel 的默认绑定。Alva Agent Channel 仍是 Web 全局 Ask、Telegram / Discord DM 和未归属消息的默认入口。

GUI 示意 · Personal Channel 连接 Telegram

Telegram 方案：

  • 默认使用同一个官方 Alva Telegram Bot。
  • 用户把 Personal Topic Channel 连接到 Telegram 后，Alva 在同一个 Telegram DM 中创建对应 topic，并保存 channel_id <-> telegram chat_id + message_thread_id 的映射。
  • Telegram 中不带 message_thread_id 的消息继续进入 Alva Agent Channel。
  • Telegram 中带 message_thread_id 的消息进入对应 Personal Topic Channel。
  • Channel 改名时可同步 Telegram topic 名称；删除 Channel 时默认只断开映射，不默认删除 Telegram topic 历史。
  • 依据：Telegram threaded conversations、createForumTopic。

Discord 方案：

  • Discord DM 没有等价的 native topic 容器；Personal Topic Channel 不做完全同步的 DM topic。
  • 短期可在同一个官方 Alva App 里用 slash command、select menu 或快捷切换指定当前 Channel。
  • 需要 native thread 体验时，用户把 Alva App 安装到 Discord server，Alva 在指定 text / forum channel 下创建 thread 映射 Personal Topic Channel。
  • 依据：Discord Threads、User-installable Apps、Application Command Contexts。

Dedicated bot 方案：

  • 不作为 Personal Topic Channel 的默认能力。
  • Telegram 可通过用户手动 BotFather 或 Telegram Managed Bots 创建专属 bot；Discord 可通过 Developer Portal 创建专属 app / bot。
  • 只用于 creator paid room、KOL channel、enterprise white-label 或强品牌隔离场景。
  • 依据：Telegram Managed Bots。

迁移规则：

  • 用户可在 Alva Agent Channel 中用自然语言创建 Personal Topic Channel。
  • 用户可把 Alva Agent Channel 中的 artifact、旧 Chat、task summary 移动到 Personal Topic Channel。
  • Start fresh in this Channel 清理该 Topic Channel 的短期上下文窗口，保留 Artifacts、Task、Memory、Skill、Settings。
  • Public read 只开放 Owner 选择公开的内容；Owner 的 private memory、portfolio、files、trading context 保持私有。

Mirrored IM Group Channel

Mirrored IM Group Channel 来自 Telegram / Discord 群，是外部群在 Alva Web 上对应的主 Channel。

GUI 示意 · Mirrored IM Group Channel

基础规则：

  • 访问角色包括 Group Admin、Bound Member、Unbound Member、Guest；Alva Agent 是群上下文代理。
  • 可见性由 Group Admin 设置，默认 private；可选 member-gated 或 public read。
  • 来源是 Telegram group 或 Discord group。
  • Alva Web Channel 是主工作面。
  • 主体验是让 Alva 记录重点、读取上下文、追踪任务、生成摘要、分发更新。

Connect IM

连接流程：

  1. Group Admin 把 Alva 加入 Telegram / Discord 群。
  2. Alva 创建或绑定对应的 Mirrored IM Group Channel。
  3. 群成员通过链接进入 Web Channel。
  4. 未绑定用户进入绑定流程。
  5. Bound Member 可提问、保存 memory、提交 source、启动 task。

连接规则：

  • 一个 Telegram / Discord 群映射一个 Mirrored IM Group Channel；Web 侧对象是权限、memory、artifacts 的主记录。
  • IM 消息作为 Channel input event 进入系统。
  • Web Channel 保存整理后的状态、Artifacts、Task、Memory 和 Skill 配置。
  • 外部 IM 成员绑定 Alva account 后获得 Web 写入能力。
  • 群接入 Alva 不改变个人资产的权限边界。

可见性和成员校验

Mirrored IM Group Channel 的 Web 内容不自动继承外部 IM 群的可见性。Alva 不能凭一个 Web 访客判断对方是路人，还是群里尚未绑定 Alva 的成员；必须先让用户证明外部 IM 身份，再校验该身份是否仍在群里。

访问状态：

  • Unknown visitor: 只有 Web session，没有 Telegram / Discord 身份；按群外路人处理。
  • Link-holder: 拿到 Web Channel 链接但未绑定；只能看到轻量 landing 和绑定入口。
  • Observed Unbound Member: Alva bot 在群里见过该 platform user id，但该身份还没有绑定 Alva account；不能给 Web 权限。
  • Bound non-member: 已绑定外部身份，但 membership check 失败；不能访问 private / member-gated 内容。
  • Bound Member: 已绑定外部身份，且 membership check 通过；可以访问 member-gated 内容。
  • Group Admin: 已绑定外部身份，且平台校验为群管理员；可以管理 visibility、成员规则和公开范围。

可见性规则：

  • Private Mirror: 默认态；只有 Bound Member 和 Group Admin 可以访问 Web Channel。
  • Member-Gated Mirror: 路人可看到轻量 landing；绑定并通过 membership check 后才能进入。
  • Public Read Mirror: 由 Group Admin 显式开启；路人只能看被标记为 public 的摘要、Playbooks、digest 或公开产物。
  • Public Read 不开放 raw chat、member list、private artifacts、task session、内部 source trace 或群内 private memory。
  • 用户离开 Telegram / Discord 群后，Web Channel 权限应失效，或进入 pending reverification。

Web 控制面和群写入

Mirrored IM Group Channel 在 Alva Web 端是群上下文控制面：用户在 Web 里和 Alva Agent 交互，查看群内沉淀，管理 Automations、Playbooks、Memory 和 Settings。

GUI 示意 · Web 控制面和显式群写入

默认规则：

  • Web 端 Chat 默认只和 Alva 交互，不把消息发回 Telegram / Discord 群。
  • 用户可以在 Web Chat 里询问群上下文、生成摘要、整理 memory candidate、修改 Automation、挂接 Playbook 或调整 digest 规则。
  • 修改 Automations、Playbooks、Memory 和 Settings 会影响未来 Alva 在群里的推送、响应和摘要，但不会自动发送一条普通群消息。
  • Bound Member 可以使用 Web 控制面；涉及群级设置、公开范围、群推送和群写入的操作需要 Group Admin 权限。

写回群规则：

  • 发回外部群必须是显式动作，例如 Post to group、Send digest now、Announce update、Publish summary、Push Playbook update。
  • 写回前必须展示 preview，并由有权限的用户确认。
  • 写回群时由 Alva bot 发送，不伪装成用户本人；消息应带轻量 attribution，例如 requested by <user> 或 configured by Group Admin。
  • Automation delivery、digest delivery 和 Group Admin confirmed publish 是群写入路径；普通 Web Chat 不是群写入路径。

群能力

Mirrored IM Group Channel 复用 Alva Agent Channel 的基础能力，但上下文范围变成群组：

  • Chat: 普通群聊进入短期上下文窗口。
  • Task: @Alva、命令、reply、显式保存、task trigger、Group Admin 规则可生成 task。
  • Artifacts: 群内生成或关联的 Playbooks、Automations、Files。
  • Memory: 群共识、关注对象、重要观点、Group Admin 规则、已验证 thesis、失效 thesis；重点是从群聊里沉淀可复用的高价值信息，不保存原始聊天记录本身。
  • Skill: Phase 3 支持群级 default skill。
  • Member: 支持 Group Admin、Bound Member、Unbound Member、Guest。
  • Settings: visibility、成员权限、公开范围、tracked targets、digest cadence、memory policy、IM 映射。

权限规则：

  • Channel access 控制频道页、main session、Artifacts、Task、Channel Memory 和 room-level context。
  • Public Channel 只公开被标记为 public 的 Channel 内容。
  • Public Channel 内的 private artifact 保持 private。
  • 单人 private context 不能进入 Mirrored IM Group Channel。
  • Mirrored IM Group Channel 的 Memory 只保存群内可见信息或 Group Admin 明确提交的信息，并保留 source message、contributor 和 last verified time。
  • 普通 Web Chat 不回写外部群；群写入必须走显式 publish / delivery 动作。
  • Task 继承 parent Channel 的访问边界。

Phase 1: Channel MVP

  • 建立 Channel-first 主体验。
  • 落地 Alva Agent Channel 作为默认主会话。
  • 新增 Personal Topic Channel 的创建、修改、删除、列表、归属和主会话。
  • 支持新用户 onboarding preset channel：按注册渠道预设 Personal Topic Channel，并挂载 Playbooks、可回看历史输出的 Automations 和参数填写 Widget。
  • 落地 Chat main session。
  • 落地 Artifacts tab，优先接入 Playbooks 和 Feed 产物；Feed 在用户端归入 Automations，再补齐 Files 的归属能力。

Phase 1.5: Legacy 兼容

  • 保留 New Chat，但不作为默认入口。
  • 保留 Chat History，但不作为 Channel IA 核心入口。
  • 旧 Chat 可保存为新 Channel，或移动到已有 Channel。
  • Task 未上线前，复杂指令在 Channel main session 中执行。
  • 复杂指令输出写入 Artifacts。

Phase 2: Task + Memory + Mirrored IM Group Channel

  • 落地 Task object。
  • 落地 Task tab。
  • 落地 Task Session，支持继续对话。
  • 支持 Task 回写 summary、artifact、memory candidate、follow-up task。
  • 将长耗时工作产品化为 Channel Task。
  • 在用户级 Memory 之外，新增 Channel scope Memory。
  • 落地 Mirrored IM Group Channel 的 IM 路由、Group Admin、群订阅和 Web 侧管理面。
  • Mirrored IM Group Channel 支持 Group Admin、Bound Member、Unbound Member、Guest。

Phase 3: Skill

  • 落地 Skill tab。
  • 支持 Channel 默认 skill。
  • 支持单次任务或对话临时使用其他 skill。

  • Task 挂载在对应 Channel 内。
  • 全局 Playbooks 作为对象库保留。
  • Channel 内 Playbooks 归入 Artifacts。
